## Headline

A self-organizing hierarchy across a robot swarm — temporarily rooted at one interchangeable "brain" robot — supplies the two architectural conditions (global state aggregation and global-vs-local actuation separation) that the authors argue make it tractable for the collective to solicit and execute externally generated code at runtime when its decentralized behavior gets stuck, with a reported 85% mission-success rate (17/20) in 34-robot simulation and a single 6-robot hardware demonstration.

## Summary

The paper extends the authors' prior **self-organizing nervous system (SoNS)** — a middleware that lets peer robots self-assemble a dynamic, rooted-graph hierarchy with a swappable root ("brain") — by adding an online loop in which the brain, on detecting that the swarm is stuck, sends its SoNS-aggregated view (obstacle and robot positions plus static context) to DeepSeek R1 via OpenRouter, receives Lua code, and propagates the updated MISSION program through the hierarchy so every robot executes it. The authors propose that two SoNS properties make this loop tractable: hierarchical separation of global from local actuation lets the LLM be asked to write a *global* swarm behavior directly rather than a local rule whose group-level effect must be reverse-engineered, and upstream sensor aggregation gives the brain an estimate of swarm-and-environment configuration with bounded staleness (max update delay n+1 steps for a rooted graph of depth n). Evidence is framed throughout as "proof-of-concept demonstrations": 17 of 20 simulation runs (25 ground + 9 aerial robots, 15 obstacles) completed the mission, and 1 hardware run with 6 robots (4 ground + 2 ceiling-tethered aerial) ran end-to-end; in all 3 simulation failures, the LLM-returned code removed the brain's stuck-detection pathway and a sub-group was left behind.

## Specifics

- **What SoNS is doing as a hierarchy** — peers form a directed rooted tree; the root (brain) is interchangeable and the structure heals as robots join or leave, so the swarm gets *temporarily centralized* coordination without losing decentralized fault-tolerance. Sensing flows upstream to the brain; actuation commands flow downstream. The brain is the only robot that talks to the LLM.
- **The two enabling properties the paper hangs its argument on** — (i) global-vs-local actuation split (`v_GLOBAL, w_GLOBAL` commanded top-down for formation-level motion; `v_LOCAL, w_LOCAL` applied per-robot for corrections like obstacle deflection), which lets the LLM write a *global* behavior directly; (ii) upstream sensor aggregation, which gives the brain an aggregated view of the swarm-and-environment with a worst-case latency of n+1 steps (n = graph depth).
- **The trigger and the loop** — each robot runs a per-step stuck-timer; when any robot's timer exceeds θ_1, the brain calls `REQUESTHELP`, which POSTs the static SoNS context (velocity components, capabilities including differential drive, sensing capabilities, known environment components, safety distance, mission goal, required functions, format example) plus the dynamic positions of detected obstacles and robots, with the open-ended prompt *"There seems to be something wrong, can you check what happened and improve my code?"* The returned Lua code is swapped into the MISSION program and propagated by each robot forwarding the new program to its children.
- **The headline number, carefully read** — 85% success is **17/20 simulation trials only**; the hardware result is n=1 demonstration, not a statistical reliability claim. Mission duration in successful runs clustered tightly around 340–349 s with a right tail to 400–409 s, reflecting genuine variation in LLM-returned strategies (the paper shows two distinct example algorithms — one deflecting only locally, one re-routing globally toward sparser-obstacle regions).
- **The failure mechanism is one mechanism, repeatedly** — all 3 unsuccessful trials failed the same way: the LLM-returned code removed the brain's mechanism for detecting that other robots were stuck, the swarm split, and a sub-group was left behind. The authors do not characterize whether the three were variants of one bug or three distinct broken paths, and the loop has no enforced separation between static (safety-relevant) code and update-able code — the LLM has latitude to overwrite anything in MISSION, including the very stuck-detection that triggers the loop.
- **What is *not* in the paper** — no ablation against a flat (non-hierarchical) swarm running the same loop; no baseline against SoNS with hand-coded obstacle avoidance; no alternate-LLM comparison. Whether the result is attributable to SoNS-the-hierarchy or to DeepSeek-R1-the-reasoner is not separable from this evidence. The authors frame the request prompt as "fairly open-ended" and the tasks as "simple," and explicitly label more principled request protocols and static/update-able code separation as future work.
- **Where this lands relative to neighboring work** — the paper distinguishes its hierarchical approach from LLM2Swarm (Strobel et al. 2024), where *individual* robots in a flat swarm self-diagnose and locally validate LLM-generated code; the argument is that a flat architecture still leaves open the problem of designing local rules whose emergent effect produces a desired group behavior, and the SoNS hierarchy resolves that by letting the LLM write the global behavior directly. No head-to-head benchmark is run.
- **Artifacts** — all 20 trial videos and the full LLM conversation logs are deposited at https://doi.org/10.5281/zenodo.17257762; this is the primary artifact for characterizing the 3 failures in depth.
