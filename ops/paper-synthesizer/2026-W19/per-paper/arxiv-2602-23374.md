## Headline
Higress-RAG introduces a four-layer enterprise-RAG stack exposed via the Model Context Protocol that orchestrates adaptive query routing, semantic caching, RRF-fused dense+sparse retrieval, and CRAG-style corrective retrieval, and reports ~+30 percentage points of Hit Rate and ~+22 percentage points of Factuality over an uncharacterized "naive baseline RAG" on two proprietary in-domain test sets.

## Summary
The paper argues that production enterprise RAG fails simultaneously on retrieval precision, hallucination rate, and real-time latency, and that the architectural response should be a "Full-Link Optimization" spanning Pre-Retrieval, Retrieval, Post-Retrieval, and Corrective Retrieval, integrated behind an MCP server interface for agent / LLM-host consumption. The defense is descriptive-engineering: every named component (Higress-Native Markdown-AST splitter, an Adaptive Router that classifies queries as Simple / Complex / External and dispatches Query Rewriter + HyDE accordingly, hybrid bge-m3 dense + BM25 sparse retrieval merged by Reciprocal Rank Fusion with k=60, a Milvus-native Metadata-Aware Boost Ranker, a bge-reranker-v2-m3 cross-encoder, a CRAG evaluator with Correct/Incorrect/Ambiguous branching to context-compression / web fallback via Tavily / merge, and a semantic cache with dynamic similarity thresholds 0.95–0.98 and 7-day TTL) is described in implementation terms and benchmarked end-to-end against a baseline. The headline empirical claims — Hit Rate of ~91% (Blog, n=611) and 92% (Docs, n=522) vs ~65% / ~60.5% baseline, Factuality of ~56.9% / 57.3% vs ~34.8% / 31.2%, cached-query latency under 50ms, uncached complex-query latency 11–20s — are reported as approximate and rest on a baseline whose retriever, reranker, generator, and prompt are not specified, on test queries that were LLM-synthesized over the same corpus the system retrieves from, and on only a cache-on/off ablation (no per-component isolation), so the paper presents the integration blueprint as the contribution rather than as new retrieval mechanics.

## Specifics
- **System framing.** Positioned as a "Modular RAG" instantiation behind MCP (Anthropic's JSON-RPC 2.0 LLM-host-to-tool standard), with the architectural pitch being vendor-agnostic agent access plus Milvus Partition Keys for multi-tenant isolation. The paper opposes "holistic" to "modular" rhetorically while operationally aligning with Modular RAG — a distinction the paper does not make precise.
- **Adaptive Router (the agentic surface).** Three routes are specified: Simple → direct vector search bypassing rewriting; Complex → full pipeline with Query Rewriter, decomposition, and HyDE (Hypothetical Document Embeddings, Gao et al. ACL 2023); External → Tavily web search bypassing the internal index. Sec VI.B concedes the External route breaks in regulated / air-gapped enterprise environments, i.e., one of the named components does not generalize to one of the named target deployments.
- **Hybrid retrieval + RRF.** bge-m3 (Chen et al. ACL Findings 2024) supplies both dense and sparse signals, merged by Reciprocal Rank Fusion with k=60. The original RRF reference (Cormack et al. 2009) is not cited.
- **Boost Ranker.** A Milvus-native, in-DB arithmetic score modifier (e.g., BoostFactor=1.2 for `doc_type=="official"`) applied before the cross-encoder reranker. The paper calls this "pivotal" for the +30% recall gain but does not isolate it in ablation.
- **CRAG layer (Yan et al. 2024).** The three-way evaluator is reproduced essentially verbatim from the CRAG paper. The scoring prompt, the LLM used for the evaluator, and the Correct/Incorrect/Ambiguous thresholds are not specified.
- **Semantic cache.** Dynamic similarity threshold (0.95 default, 0.98 for fuzzy/uncertain queries), 7-day TTL; cache-on/off is the only direct ablation reported and is what supports the <50ms latency claim. Staleness control is asserted but not directly tested.
- **Evaluation caveats (load-bearing for how to read the numbers).** (1) "Baseline RAG" is unspecified — retriever, reranker, generator, prompt all unstated. (2) Factuality Score measures answer-context consistency, not answer-truth — the paper itself describes it as a hallucination *proxy*, and a system that retrieves and parrots context can score well even when the context is wrong (abstract-only-style caveat applies even with full text). (3) Test queries on both datasets (Higress Blog n=611, Higress Docs n=522) were generated by an automated QA pipeline over the same corpus the system retrieves from, which inflates Hit Rate vs natural user queries and risks circularity. (4) No public benchmark, no third-party domain, no error bars, no significance tests, no seeds. (5) The "30% improvement" is described as approximate; the actual deltas are +25.8pp on Blog and +31.5pp on Docs.
- **Hedging preserved.** The paper claims and asserts; it does not "prove." D1 (full-link beats modular) is *claimed but not directly tested*. D4 (Boost Ranker pivotal) is *claimed* but not isolated. D5 (CRAG reduces hallucination) is supported by a *proxy* metric. D7 (MCP solves vendor lock-in) is *asserted as an architectural property*, not empirically tested.
- **Generation LLM.** Qwen3-8B (8B-parameter), pinned to GPU 1, port 8002.
- **Provenance flags.** Single-author arXiv preprint, no DOI, header decoratively names "IEEE Transactions on AI" though not published there. Reference list has six items total and omits the original RRF paper, BM25, the Naive/Advanced/Modular RAG taxonomy source, and any prior enterprise-RAG system it positions itself against.
