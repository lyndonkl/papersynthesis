### One-line summary

`RAGSearch` benchmarks dense RAG against GraphRAG under agentic-search conditions and finds that agentic search narrows the gap on simpler tasks but GraphRAG retains an edge on complex multi-hop reasoning.

## Pass 2 — Content grasp

### Main argument and supporting evidence

The paper argues that the rise of agentic search (iterative retrieval-reasoning loops, optionally trained with RL) does **not** make GraphRAG's explicit graph structure obsolete: agentic interaction supplies *implicit* structure that compensates for missing graph structure on simple/general QA, but *explicit* graph structure remains "crucial for robust multi-hop reasoning." To support this, the authors build `RAGSearch`, a unified benchmark that decouples the agentic control layer (the ReAct-style think-search-answer loop) from the retrieval backend (dense RAG or one of five representative GraphRAGs — GraphRAG, RAPTOR, HippoRAG2, HyperGraphRAG, LinearRAG), holds the agent and inference budget fixed, and runs six QA datasets (NQ, PopQA, TriviaQA for general QA; HotpotQA, 2Wiki, Musique for multi-hop). They show (Table 1) that under static single-shot inference GraphRAG beats dense RAG by ~+27.23 points Contain-EM on multi-hop QA but by only +0.47 on general QA; under training-free agentic search (GraphSearch) the multi-hop Graph–Dense gap shrinks from +27.23 to +26.59 (a 32.3% reduction relative to the second-best variant) while general-QA gains for agentic search appear mainly when paired with dense RAG; and under RL-based agentic training (Search-R1 / Graph-R1, GRPO) graph backends — especially entity-centric HippoRAG2 — still dominate on multi-hop. Tables 2–3 and Figures 3–4 further establish that GraphRAG yields lower variance and higher recall under iterative agentic search, while Table 8 documents that GraphRAG retains a substantial offline preprocessing-cost overhead (e.g., GraphRAG itself at 1.72h/1M tokens, $13.19/1M tokens, 22,160 context tokens per query — versus dense RAG's near-zero offline cost). The paper's hedging is preserved: agentic search "partially mitigates" the absence of explicit structure on multi-hop tasks; it does not eliminate the gap.

### The Big Question

How should retrieval-augmented LLM systems organize external knowledge — through explicit pre-built relational structure (GraphRAG-style entity/hierarchy/hypergraph indexing) or through implicit structure that emerges at inference time via agentic multi-round retrieval-and-reasoning? More concretely: in the new design space where retrieval is dynamic and decision-making is sequential, does the expensive offline graph-construction step still earn its keep?

### Specific hypotheses tested

The paper frames these as research questions (RQ1-RQ5); they function as the testable claims:

1. **RQ1 — Substitution hypothesis:** Agentic search can compensate for the absence of explicit graph structure in dense RAG. Tested by comparing dense-vs-graph backends across single-shot, training-free agentic (Search-o1, GraphSearch), and RL-based agentic (Search-R1, Graph-R1) regimes (Table 1).
2. **RQ2 — Persistence-under-training-free hypothesis:** Even under training-free agentic search, explicit graph structure continues to provide benefits. Tested by varying retrieval backends within a fixed agentic pipeline (Table 2).
3. **RQ3 — Backend-dependence-of-RL hypothesis:** RL-based policy learning interacts heterogeneously with different retrieval infrastructures (i.e., RL gains depend on which graph backend it is trained against). Tested via Graph-R1 ablations across five GraphRAG backends (Figure 3, Table 1).
4. **RQ4 — Robustness/stability hypothesis:** GraphRAG yields more robust and stable behavior under agentic search (higher recall, lower Contain-EM variance). Tested via sensitivity analysis with mean ± variance over multiple runs (Table 3).
5. **RQ5 — Module-impact hypotheses:** (a) GRPO is a favorable RL training paradigm for agentic RAG (compared to PPO, RPP) and (b) larger LLM backbones partially close the Graph–Dense gap. Tested in Figure 4 and Tables 4–5.

### Unfamiliar terms requiring gloss

- **Agentic search / agentic RAG** — Retrieval invoked iteratively inside an LLM's reasoning loop (think → search → information → think → ... → answer), as opposed to "single-shot" / "static" RAG where one retrieval call precedes generation. Formalized here as a ReAct-style interleaved loop with `<think>`, `<search>`, `<information>`, `<answer>` tags.
- **GraphRAG (family)** — RAG variants that organize the corpus as a graph (entities + relations / hierarchies / hyperedges) and retrieve nodes, paths, or subgraphs as evidence, rather than retrieving free-text chunks by semantic similarity alone.
- **Tree-based / entity-graph / hypergraph / tri-graph GraphRAG** — Four sub-families of GraphRAG that differ in the index structure: hierarchical clustering trees (GraphRAG, RAPTOR), entity-centric graphs (HippoRAG2), hyperedges over n-ary facts (HyperGraphRAG), and relation-free lightweight Tri-Graphs built from entity extraction + semantic linking (LinearRAG).
- **HippoRAG2** — Memory-inspired entity-centric GraphRAG that applies Personalized PageRank–style propagation over an entity graph to retrieve multi-hop evidence.
- **Search-o1 / GraphSearch / Search-R1 / Graph-R1** — The paper's four representative agentic systems: Search-o1 (training-free reasoning-driven on-demand search with a Reason-in-Documents module), GraphSearch (training-free orchestrated multi-agent workflow: decomposition → search → verification → answer), Search-R1 (RL-trained dense-RAG agent), Graph-R1 (RL-trained GraphRAG agent).
- **GRPO (Group Relative Policy Optimization)** — The RL algorithm used for trajectory-level optimization of the agent policy, sampling K trajectories per query and using the group-mean-normalized reward as advantage, with KL regularization to a reference policy. Inherited from DeepSeek-Math / Search-R1.
- **Contain Exact Match (Contain-EM)** — The paper's primary metric: whether the normalized ground-truth answer appears as a substring of the normalized generated response. Used alongside F1 (token overlap).
- **Reasoning-driven on-demand search vs orchestrated multi-agent workflows** — Two patterns the paper distinguishes within training-free agentic RAG: the first triggers retrieval reactively based on the model's own uncertainty (Search-o1); the second decomposes the query a priori into a fixed pipeline of sub-modules (GraphSearch).
- **Knowledge Refinement / Reason-in-Documents** — Search-o1's module that compresses each retrieval result into a concise summary before injecting it into the agent state, to fit long-horizon reasoning into the context window.
- **Train-free vs train-based agentic search** — Whether the agent's control policy is purely prompt-defined (train-free) or fine-tuned via RL on the agent–environment trajectory (train-based).

### Figure-by-figure analysis

- **Figure 1 (Explicit vs Implicit Structure in RAG Systems).** Schematic. Top row shows GraphRAG building a KG offline from documents, then retrieving over it. Bottom row shows dense RAG (text-chunk embeddings) and beneath that an "Agentic Search" sub-panel where the LLM iterates queries against a knowledge base. Takeaway is conceptual — it frames the paper's central contrast: explicit (precomputed, structural) vs implicit (emergent, interactive) structure. No quantitative content; serves as a thesis diagram.
- **Figure 2 (Overview of RAGSearch).** Timeline strip across the top maps GraphRAG sub-families to their first-publication dates (Tree 2024.04 → Entity-graph 2025.02 → Hypergraph 2025.03 → Tri-graph 2025.10). The lower panels diagram the three benchmark regimes: General Search (single-shot), Training-free Paradigm (think → search action → think → answer), and Agentic Search with optional RL Training (policy LLM with Reward Manager combining Format Reward + Outcome Reward). Takeaway: the benchmark is structured as a 3-regime × 6-backend × 6-dataset grid.
- **Table 1 (Overall Contain EM, single-shot vs training-free vs RL).** The headline result table. Rows are systems (Qwen-2.5-7B + Dense / GraphRAG★, Search-o1 + Dense / GraphRAG★, GraphSearch + Dense / GraphRAG★, Search-R1, Graph-R1★); columns are NQ / PopQA / TriviaQA (General QA) and HotpotQA / 2Wiki / Musique (Multi-hop). Arrows show graph-vs-dense delta on each cell. Load-bearing observations: (i) single-shot GraphRAG gains on multi-hop are large (+27.70, +27.03, +26.99 on HotpotQA / 2Wiki / Musique) but small or negative on general QA (+1.69 NQ, +0.68 PopQA, −0.95 TriviaQA); (ii) under GraphSearch agentic, dense RAG itself jumps substantially (e.g., NQ 38.20 → 58.27, HotpotQA 33.76 → 38.22) but GraphRAG paired with GraphSearch still leads on multi-hop (e.g., HotpotQA 58.64 vs 38.22 = +20.42); (iii) RL-based Graph-R1 trails some training-free GraphSearch+graph configurations, the paper's basis for "well-designed training-free workflows remain highly competitive and can even exceed RL-based systems." n is not stated per cell, but full test/dev sets are used per Table 6; the ★ marks the best-performing GraphRAG variant per row. Error bars are not reported in Table 1.
- **Table 2 (Training-free agentic systems × six retrieval backends, Search-o1 and GraphSearch).** This is the key RQ2 evidence. For each agentic method, rows enumerate the six backends (Dense / Hypergraph / HippoRAG2 / LinearRAG / RAPTOR / GraphRAG) and report per-dataset scores plus Avg Rank within method. Takeaway: HippoRAG2 (entity-centric) achieves Avg Rank 1.00 under Search-o1 and 1.33 under GraphSearch on multi-hop; dense RAG sits at Avg Rank ≈ 5–6 on multi-hop in both regimes. On General QA, dense RAG is competitive (Avg Rank 2.33 under Search-o1, 4.00 under GraphSearch). The asymmetry between multi-hop and general QA is the cleanest signal in the paper.
- **Table 3 (Sensitivity Analysis: search turns, recall, mean ± variance Contain-EM on HotpotQA and PopQA).** Reports Search-o1-Dense vs Search-o1-HippoRAG2 and Search-R1 vs Graph-R1-HippoRAG2. Variances are reported (e.g., HippoRAG2 variance 0.22 on HotpotQA vs 1.03 for dense); search turns are also reported (Graph-R1-HippoRAG2 uses fewer turns: 1.71 on HotpotQA vs 1.82 for Search-R1). This is the load-bearing evidence for RQ4 (robustness/stability). Sample sizes for the variance estimates are not specified in-table.
- **Figure 3 (Performance of different retrieval infrastructures on RL-based agentic systems).** Radar plot across six datasets (HotpotQA, NQ, Musique, PopQA, TriviaQA, 2Wiki) for five Graph-R1 variants plus Search-R1-Dense. Visual takeaway: graph backends form a larger envelope on multi-hop axes; dense holds its own on NQ. Hard to read exact numbers, but the qualitative claim is supported.
- **Figure 4 (RL algorithm impact: GRPO vs PPO vs RPP on Graph-R1 with three backends).** Three sub-panels (HypergraphRAG-3B, LinearRAG-3B, HippoRAG2-3B) plot Accuracy and F1 across HotpotQA / NQ / Musique / 2Wiki. ### Confusions / unresolved points

1. **Promotion of "32.3% reduction" claim.** §5.2.2 reports the multi-hop Dense–GraphRAG gap "decreases from +27.23 (single-shot) to +26.59" — a delta of just 0.64 points. The same sentence then asserts this is "reduced by 32.3% relative to the second-best GraphRAG variant." The 32.3% figure is not derived in-line and the denominator is not explicit; readers cannot trace the calculation without examining auxiliary numbers. This is a hedging-vs-evidence tension: the absolute reduction (0.64) is modest, but the relative framing makes it sound large.
2. **Sample sizes and seeds.** Tables 1–2 report point estimates without per-cell confidence intervals. Only Table 3 reports variance (mean ± SD), and only for two datasets and two pairs of systems. The claim "GraphRAG is more robust and stable" (Observation 6) rests on this narrow slice. Number of seeds per cell is not stated.
3. **Conference template artifacts in the metadata.** The header still reads "Conference acronym 'XX, June 03–05, 2018" and "Trovato et al." in the running head, and the CCS Concepts section literally reads "Do Not Use This Code → Generate the Correct Terms for Your Paper." These are unfilled ACM template placeholders — they do not impact the technical claims but signal the manuscript was not fully cleaned before posting.
4. **Cost comparison asymmetry.** Table 8 reports GraphRAG offline costs but no comparable line for dense RAG (presumably zero offline, but online retrieval cost is also relevant for fair comparison and is not tabulated). Without dense's online cost in the same table, the "GraphRAG is more expensive but worth it for multi-hop" trade-off is supported on the construction side only.
5. **"Crucial for multi-hop reasoning" vs "narrows the gap."** The Conclusions section calls explicit structure "crucial" for multi-hop. Yet under GraphSearch+Dense (Table 1), dense agentic search itself jumps to HotpotQA = 38.22 (compared to 33.76 single-shot) — a meaningful gain. "Crucial" reads stronger than the evidence; "remains beneficial and stable" would match Tables 1–3 more cleanly.
6. **Backbone-scale generalization.** The 32B comparison is only run in the training-free Search-o1 regime (Table 5), not for RL-based methods. Whether RL-based agentic graph advantages also shrink at 32B is left open.
7. **Bing Web Search substitution (Appendix B Prompting Details).** Search-o1 was originally designed for live web search; here, "the actual Bing Web Search API was replaced with a retrieval server" returning Wikipedia chunks "formatted to mimic web search results." This is a defensible methodological choice for controlled benchmarking but means the Search-o1 numbers are *not* directly comparable to Search-o1's published results.
8. **Single LLM family.** All experiments use Qwen2.5 variants. Whether the Dense–Graph gap closure under agentic search holds for other model families (Llama-3, GPT-4, Mistral) is not tested.
9. **No statistical-significance tests.** Differences as small as +0.09 (Search-o1-7B-Dense vs Search-o1-7B-GraphRAG★ on NQ, Table 1) are reported alongside differences of +27.70, but the paper does not distinguish noise-level from substantively-large deltas.
